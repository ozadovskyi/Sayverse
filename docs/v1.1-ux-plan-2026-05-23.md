# v1 UX scope expansion — 2026-05-23

Captures the design decisions for the UX work landing **before** the App Store submission, not after. Reordered 2026-05-23: the user's call is that v1 should ship with this UX in place rather than re-iterating right after launch — better first impression, no separate v1.1 review cycle.

Inputs that fed into this doc: a 6-point beta-tester feedback received 2026-05-23, two rounds of competitive UX research (mode split + Quick Translate surface), and a re-prioritisation of the existing v2 backlog.

## Beta-tester feedback (2026-05-23)

Verbatim themes from the tester, condensed:

1. Replace the "Speak what type to get translator" placeholder during translation with an obvious loading state (spinner + "Translating…").
2. Auto-scroll to the **start** of the new translation, not to the bottom of the conversation. Critical for long translations that overflow one screen.
3. UI behaves inconsistently between modes: single renders source + translation as two separate cards; conversation merges both into one bubble. Either unify or make the difference explicit.
4. Add a setting to hide the original text and show only the translation — especially useful in conversation mode.
5. Reconsider the Single vs Conversation split. Functionality overlaps; mode boundary is unclear. Either merge, or make Conversation default and clearly differentiate Single.
6. "New conversation" and "History" buttons sit too close to the main record control — mis-tap risk on small screens.

## Research findings — mode split (single vs conversation)

Question: should we merge the two modes, keep both with conversation default, or keep as-is?

- **Every leading translator ships two distinct surfaces.** Google Translate, Apple Translate, Microsoft Translator, DeepL, iTranslate all have separate "Conversation" entries distinct from single-translate. iTranslate Converse is sold as a wholly separate product/SKU.
- **The 2025–2026 trend is reinforcing the split**, not collapsing it: Google + Gemini Conversation redesign, DeepL Voice for Conversations (new product 2025), Apple Live Translation as another distinct surface in iOS 26.
- **The only app that merged the two (SayHi) was discontinued in 2024.** Users migrated to Microsoft Translator, which keeps them separate.
- **Review aggregators show no significant "I confuse the two modes" complaints.** Complaints about conversation mode exist (quality, mid-sentence cutoffs), but not about its existence as a distinct surface.
- The conventional UX framing: single optimises for "translate this sign / phrase to send", conversation optimises for face-to-face turn-taking with split layout, auto-detect, and a running transcript. **Different jobs-to-be-done.**

**Decision**: keep both, make Conversation the default, visually re-differentiate Single so the tester's perceived redundancy is resolved as a labeling / layout problem (which it actually is), not an architectural one. Per-app citations in the agent transcript dated 2026-05-23.

## Research findings — Quick Translate surface

Conventional layout shared by Google / Apple / DeepL / Microsoft:

- Source ⇄ target language chips at the top, swap button between them.
- Text input area below, multi-line, with mic button **inside** the field as a trailing icon and a clear (×) button.
- Result card below the input, with three persistent action buttons: Copy, Speak (TTS), Share.
- Recent history visible inline on the main screen (last ~10), not buried in a separate screen. Reviews repeatedly cite this as table stakes.
- Star / favourites on each history row.

Top user-review complaints to design against:

- Swap languages wiping the typed text (Microsoft Translator review threads).
- Long translations un-copyable because selection scroll breaks.
- TTS auto-play startling users in public.
- 38% of users unaware of features that exist — discoverability is the dominant problem (Pendar Yousefi, Google Design).

Input preferences from the limited public data: tourists / travellers lean voice + camera; learners and professionals lean type + paste. A voice-only Quick Translate is a wrong default.

## Resolved open questions

### Voice input — what shows while the user is speaking?

**Decision: live transcript only, no live translation.** The transcript appears in the input area token-by-token via `gpt-4o-transcribe` streaming. Translation fires once after the user stops recording.

Reasoning:
- The VUI rule "2-3 seconds without visual feedback causes frustration" is satisfied by the transcript alone — no incremental translation needed.
- Mid-sentence partial translations are low quality (missing context) — this is exactly what Google's continuous translation mode is criticised for.
- Cost: one translation call vs many partial ones, where the partial calls produce inferior output anyway. With BYOK, this is non-trivial money.

### Typed input — translate as the user types?

**Decision: no. Explicit Translate button only.** No "auto-translate as I type" setting, no toggle.

Reasoning:
- Not in the tester feedback — no user explicitly requested it.
- In broader review research it appears as a "wow"/praised pattern in Google and DeepL, but never as an explicit ask. Users notice when it's there; they don't complain when it's not.
- BYOK makes it actively bad: every debounce window costs an OpenAI call. Typing a sentence = 5–10 incremental partial translations for one mental input.
- Incomplete sentences translate poorly (same context problem as voice).
- The deliberate "press Translate" beat is a cleaner UX than constant background API spend.

## Scope

Prioritised. Each item links back to a tester feedback point (T#) or research recommendation (R).

### Group A — UX polish (do first, ~1 small PR)

- **A1 — Loading state during translation** (T1). Replace the placeholder with "Translating…" + a small spinner / pulse animation. If A4 (streaming) lands in the same PR, A1 is largely obviated by the first token arriving — keep a brief spinner for the gap between request fire and first token.
- **A2 — Auto-scroll to start of new translation** (T2). `scrollTo(messageY)`, not `scrollToEnd`. Applies in conversation mode where the new turn may not fit on one screen.
- **A3 — "New conversation" + "History" relocation** (T6). Move out of the bottom record-control zone and into the header (next to SETTINGS). Reduces mis-tap risk; matches the existing wordmark + HISTORY + SETTINGS row already proven through device testing.

### Group B — Streaming (do second, separate PR)

- **B1 — Streaming translation** (carried forward from `device-test-findings-2026-05-22.md` item 9). OpenAI streaming API + progressive `TranslationCard` rendering. Touches `services/openai.ts`, `App.tsx`, `useConversation`. The slide-up animation in `TranslationCard` / `ConversationView` must trigger on mount only, not on each text change, or it will fire on every token.
- **B2 — Live voice transcript** — **deferred to a separate effort post-v1**. Originally bundled with B1; split off after a scope review. Reasons: (1) streaming Whisper requires switching from `whisper-1` to `gpt-4o-mini-transcribe` + `stream: true`, which only works with SSE parsing on raw fetch (the SDK + React Native FormData combination is the same blocker the current code already works around); (2) the `verbose_json` response is no longer available, so we lose the per-segment `no_speech_prob` / `avg_logprob` hallucination detection that powers the silence gate (`isNonSpeechTranscription`) — that's a real regression we'd have to replace; (3) `whisper-1` returns the detected language directly in the response, the streaming endpoint does not, so language detection has to move into a separate `detectLanguage(transcript)` round-trip, weakening the auto-direction-routing that single + conversation modes rely on. The perceived-latency win is also modest — transcription on short clips already returns in ~1-2 s. The B1 translation-streaming win was the larger half of the user-visible perceived-latency gain; B2 stays on the backlog as a follow-up once we either find an SDK path that works in RN or are willing to do the SSE plumbing ourselves.

### Group C — Mode re-differentiation (do third, ~1 PR)

The architectural answer to T3 + T5. Resolves them via re-labeling and layout, not by merging the modes.

- **C1 — Conversation = default surface** (R). First-launch state and after-restart state is Conversation. Single requires an explicit tap to enter.
- **C2 — Rename "Single" to "Quick Translate"** (R). Updates copy + the toggle label so the two surfaces read as distinct tools, not "two modes of the same thing".
- **C3 — Quick Translate layout pass** (R). Bring the surface to the conventional pattern:
  - Language picker + swap button at top (swap **must** preserve both source and result texts — common bug to avoid).
  - Input area in the middle with mic as a trailing icon inside the input field and a clear (×) button.
  - Result card below input with persistent Copy + Speak (TTS) + Share icon buttons. Copy already shipped in PR #29 — verify Speak and Share land here. TTS is **tap-to-play, never auto-play**.
  - Inline "Recent translations" list (last 5–10) on the main screen, sourced from the existing unified history (PR #29). Tap a row → restores both source and result into the active state.

### Group D — Settings (small follow-up, can ride C or A)

- **D1 — Hide-original toggle** (T4). Settings option. In conversation mode, render only the translated side of each turn when enabled. In Quick Translate, the source input is still shown (the user is typing into it); the result card displays translation only without a labeled "Original" preview.

## Out of scope / dropped from earlier backlog

Re-evaluated against today's tester feedback + research + positioning pivot:

- **Unified history** (was a v2-backlog item) — **already shipped in PR #28** (single + conversation merged into one History screen with a `SINGLE` / `CONVERSATION` type marker per entry). Mention here only to mark obsolete.
- **Circuit-routing EdgeTrail** (was v2 polish) — **dropped**. The underlying complaint ("trail does not visibly thread the pill") was solved differently by the PR #30 anchor work via `measureInWindow`. The original problem no longer exists, and the circuit-routing variant adds tuning cost without product value.
- **Whisper transcription-accuracy eval** (was v2 LLM-eval category 6) — **drop from active backlog**. After the positioning pivot to AI-PM / TPM, the marginal portfolio value of a 6th eval category is low; the existing 5 categories + Maestro voice path are sufficient signal. Leave a one-line note in `llm-evaluation.md` for completeness.
- **Visual-regression testing (Vision-AI platforms)** — **stay deferred**. The earlier reasoning still holds: animated `EdgeTrail` is the documented weak case even for Vision-AI; ROI is wrong for a solo v1; paid platform adoption is not justified. Documented decision, not an oversight.
- **AI-exploratory / agentic testing (Maestro MCP)** — **stay in backlog**, lower priority. Forward-looking portfolio signal post-v1.1; not blocking.

## Implementation order

All four groups land **before** the App Store submission. Sequenced for risk (smallest / safest first):

1. **Group A — UX polish PR.** Smallest, lowest risk, addresses 3 of 6 tester points. Loading state, auto-scroll, button relocation. Good warm-up before structural changes.
2. **Group B — Streaming + live transcript PR.** Perceived-latency win; bundled because both items restructure the audio + result path around the OpenAI streaming consumer.
3. **Group C — Mode re-differentiation PR.** Most structural change (navigation, default surface, layout pass). Land after A and B so the Quick Translate layout can already display streaming results + the new loading state.
4. **Group D — Settings follow-up.** Trivial; can ride A or C depending on what's open at the time.
5. Device-test the bundled UX on iPhone 12 mini against the same checklist that produced PR #28 and PR #29.
6. **Then** App Store production submit.

Estimated total effort across A–D: **~12–18 h**, dominated by Group B (streaming pipeline restructure) and Group C (Quick Translate layout pass). Add ~1–2 h for the device-test pass before submit.
